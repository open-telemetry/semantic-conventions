# Semantic conventions for HTTP client network timing

**Status**: [Development][DocumentStatus]

This document defines the `http.client.network_timing` event for **HTTP client**
instrumentations that can report per-phase network timings on outbound requests—for example
on **mobile** platforms or other client environments where the network stack exposes phase
boundaries (DNS, TCP connect, TLS, request/response transfer).

The event complements [HTTP client spans](http-spans.md): the span records the overall client
request; this event adds elapsed time at individual network phases when the platform provides them.
Timing attribute definitions are in the
[HTTP Client Network Timing Event Attributes](../registry/attributes/http.md#http-client-network-timing-event-attributes)
registry section.

The autogenerated event section below is the complete normative attribute list (sorted alphabetically). The [Network timing attributes](#network-timing-attributes) and [Opt-in attributes from the HTTP client span](#opt-in-attributes-from-the-http-client-span) subsections group the same attributes for reading.

**Status:** ![Development](https://img.shields.io/badge/-development-blue)

The event name MUST be `http.client.network_timing`.

This event captures per-phase timing for an HTTP client request.

This event MAY be emitted when instrumentation has access to network phase timings for an HTTP client call.

**Timestamp:** MUST be the absolute time when the HTTP client call started, in milliseconds since the Unix epoch.

**Timing attributes:** All timing attributes on this event (including `http.call.end_time`) MUST be interpreted as non-negative deltas in milliseconds from `Timestamp` (elapsed time from call start to each recorded phase boundary).

**ObservedTimestamp:** SHOULD be set to the time the event was emitted when it differs materially from `Timestamp`.

**Trace context:** When an OpenTelemetry trace is available for the operation, the event MUST use the Context of the corresponding HTTP client span so backends can correlate the event with the span. When no such span exists, the event MAY omit span correlation.

**Attribute emission:** Instrumentations SHOULD emit only timing attributes that the platform provides and SHOULD omit attributes that are unknown or not applicable.

**Span attribute copy:** Instrumentations MAY expose a configuration boolean to copy a curated list of attributes from the corresponding HTTP client span onto this event. Copied attributes MUST use the same semantics as on the [HTTP client span](/docs/http/http-spans.md); attribute definitions are in the attribute registry (see [HTTP client span](/docs/http/http-spans.md) and linked registry pages). On this event, copied attributes are `Opt-In` when the option is enabled (requirement levels may differ from the span). Those attributes support metrics filtering and aggregation on backends that cannot yet correlate by context alone. When the option is disabled, those attributes SHOULD be omitted. When it is enabled, instrumentations SHOULD populate them.

**[1] `exception.type`:** If the recorded exception type is a wrapper that is not meaningful for
failure classification, instrumentation MAY use the type of the inner
exception instead. For example, in Go, errors created with `fmt.Errorf`
using `%w` MAY be unwrapped when the wrapper type does not help
classify the failure.

**[2] `http.request.method`:** HTTP request method value SHOULD be "known" to the instrumentation.
By default, this convention defines "known" methods as the ones listed in [RFC9110](https://www.rfc-editor.org/rfc/rfc9110.html#name-methods),
the PATCH method defined in [RFC5789](https://www.rfc-editor.org/rfc/rfc5789.html)
and the QUERY method defined in [httpbis-safe-method-w-body](https://datatracker.ietf.org/doc/draft-ietf-httpbis-safe-method-w-body/?include_text=1).

If the HTTP request method is not known to instrumentation, it MUST set the `http.request.method` attribute to `_OTHER`.

If the HTTP instrumentation could end up converting valid HTTP request methods to `_OTHER`, then it MUST provide a way to override
the list of known HTTP methods. If this override is done via environment variable, then the environment variable MUST be named
OTEL_INSTRUMENTATION_HTTP_KNOWN_METHODS and support a comma-separated list of case-sensitive known HTTP methods.

![Development](https://img.shields.io/badge/-development-blue)
If this override is done via declarative configuration, then the list MUST be configurable via the `known_methods` property
(an array of case-sensitive strings with minimum items 0) under `.instrumentation/development.general.http.client` and/or
`.instrumentation/development.general.http.server`.

In either case, this list MUST be a full override of the default known methods,
it is not a list of known methods in addition to the defaults.

HTTP method names are case-sensitive and `http.request.method` attribute value MUST match a known HTTP method name exactly.
Instrumentations for specific web frameworks that consider HTTP methods to be case insensitive, SHOULD populate a canonical equivalent.
Tracing instrumentations that do so, MUST also set `http.request.method_original` to the original value.

**[3] `network.protocol.version`:** If protocol version is subject to negotiation (for example using [ALPN](https://www.rfc-editor.org/rfc/rfc7301.html)), this attribute SHOULD be set to the negotiated version. If the actual protocol version is not known, this attribute SHOULD NOT be set.

**[4] `server.address`:** When observed from the client side, and when communicating through an intermediary, `server.address` SHOULD represent the server address behind any intermediaries, for example proxies, if it's available.

**[5] `server.port`:** When observed from the client side, and when communicating through an intermediary, `server.port` SHOULD represent the server port behind any intermediaries, for example proxies, if it's available.

**[6] `url.full`:** For network calls, URL usually has `scheme://host[:port][path][?query][#fragment]` format, where the fragment
is not transmitted over HTTP, but if it is known, it SHOULD be included nevertheless.

`url.full` MUST NOT contain credentials passed via URL in form of `https://username:password@www.example.com/`.
In such case username and password SHOULD be redacted and attribute's value SHOULD be `https://REDACTED:REDACTED@www.example.com/`.

`url.full` SHOULD capture the absolute URL when it is available (or can be reconstructed).

Sensitive content provided in `url.full` SHOULD be scrubbed when instrumentations can identify it.

![Development](https://img.shields.io/badge/-development-blue)
Query string values for the following keys SHOULD be redacted by default and replaced by the
value `REDACTED`:

* [`AWSAccessKeyId`](https://docs.aws.amazon.com/AmazonS3/latest/userguide/RESTAuthentication.html#RESTAuthenticationQueryStringAuth)
* [`Signature`](https://docs.aws.amazon.com/AmazonS3/latest/userguide/RESTAuthentication.html#RESTAuthenticationQueryStringAuth)
* [`sig`](https://learn.microsoft.com/azure/storage/common/storage-sas-overview#sas-token)
* [`X-Goog-Signature`](https://cloud.google.com/storage/docs/access-control/signed-urls)

This list is subject to change over time.

Matching of query parameter keys against the sensitive list SHOULD be case-sensitive.

![Development](https://img.shields.io/badge/-development-blue)
Instrumentation MAY provide a way to override this list via declarative configuration.
If so, it SHOULD use the `sensitive_query_parameters` property
(an array of case-sensitive strings with minimum items 0) under
`.instrumentation/development.general.sanitization.url`.
This list is a full override of the default sensitive query parameter keys,
it is not a list of keys in addition to the defaults.

When a query string value is redacted, the query string key SHOULD still be preserved, e.g.
`https://www.example.com/path?color=blue&sig=REDACTED`.

## Examples for derived metrics

> [!NOTE]
>
> This section is **non-normative**. It lists **possible** ways to derive durations or intervals
> from `http.client.network_timing` attributes for dashboards or downstream metrics. Actual definitions
> of metrics in OpenTelemetry semantic conventions may differ; formulas assume all referenced
> attributes are present and that deltas share the same time origin (the event `Timestamp`).

| Derived measure (example) | Possible calculation |
| --- | --- |
| DNS resolution duration | `http.dns.end_time` − `http.dns.start_time` |
| TCP connection duration | `http.connect.end_time` − `http.connect.start_time` |
| TLS handshake duration | `http.secure_connect.end_time` − `http.secure_connect.start_time` |
| Total network setup duration | `http.connect.end_time` − `http.dns.start_time` |
| Total request transmission duration | `http.request.body.end_time` − `http.request.headers.start_time`, or `http.request.headers.end_time` − `http.request.headers.start_time` when there is no body |
| Total response reception duration | `http.response.body.end_time` − `http.response.headers.start_time`, or `http.call.end_time` − `http.response.headers.start_time` when response body timings are unavailable |
| Total network transfer duration | `http.response.body.end_time` − `http.request.headers.start_time`, or `http.call.end_time` − `http.request.headers.start_time` when body end time is unavailable |
| Time to first byte (TTFB) | `http.response.headers.start_time` − `http.request.body.end_time`, or `http.response.headers.start_time` − `http.request.headers.end_time` when there is no body |
| Time to last byte (TTLB) | `http.response.body.end_time` − `http.request.body.end_time`, or `http.call.end_time` − `http.request.headers.end_time` when body timings are incomplete |
| Total request duration (same origin as deltas) | `http.call.end_time` (elapsed from event `Timestamp` to call complete) |
